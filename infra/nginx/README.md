# nginx configs — BookReady origin (root@198.211.116.44)

Configuration for the DigitalOcean origin server that sits behind
Cloudflare. Three Cloudflare-fronted hosts:

| Host                  | Upstream                                          |
| --------------------- | ------------------------------------------------- |
| `app.bkrdy.me`        | Next.js via pm2 (`bookready-web`) on `:3000`      |
| `api.bkrdy.me`        | Laravel via PHP-FPM, doc root `api/public/`       |
| `*.bkrdy.me`          | Same Next.js — tenant slug resolved from `$host`  |

## Why this exists

Before this directory landed, the origin nginx accepted any TCP connection
on `:80` / `:443` regardless of source IP. An attacker who knew the
origin IP (`198.211.116.44`, trivially recoverable from DNS history /
certificate transparency / a single old curl) could bypass Cloudflare's
WAF, rate-limit, and DDoS protection by hitting the IP directly with
a forged `Host: api.bkrdy.me` header. Verified vulnerable on 2026-05-27:

```
$ curl -sI -k -H "Host: app.bkrdy.me" \
    --resolve app.bkrdy.me:443:198.211.116.44 \
    https://app.bkrdy.me/login
HTTP/1.1 200 OK
Server: nginx/1.24.0 (Ubuntu)
...
```

These configs lock the origin so it only accepts traffic from
Cloudflare's published edge IP ranges, with the visitor's real IP
recovered from `CF-Connecting-IP` so per-IP Laravel throttles still work.

## Layout

```
infra/nginx/
├── bin/
│   └── refresh-cf-ips.sh              → /usr/local/bin/refresh-cf-ips.sh
├── snippets/
│   ├── cloudflare-realip.conf         → /etc/nginx/snippets/cloudflare-realip.conf
│   └── hsts.conf                      → /etc/nginx/snippets/hsts.conf
└── sites/
    ├── app.bkrdy.me.conf              → /etc/nginx/sites-available/app.bkrdy.me
    ├── api.bkrdy.me.conf              → /etc/nginx/sites-available/api.bkrdy.me
    └── tenant-wildcard.bkrdy.me.conf  → /etc/nginx/sites-available/tenant-wildcard.bkrdy.me
```

`/etc/nginx/snippets/cloudflare-geo.conf` is **not** in version control
because it's auto-generated by `refresh-cf-ips.sh` from Cloudflare's
published IP lists. The first run on a fresh server creates it.

### Why `geo` and not `allow`/`deny`

The realip module rewrites `$remote_addr` to the `CF-Connecting-IP`
header value **before** the access-control phase runs, so an
`allow`/`deny` block would always check the visitor's real IP
against Cloudflare's ranges and return 403 for every legitimate
request. Confirmed the hard way 2026-05-27 during the first deploy.

The fix uses a `geo` block keyed on `$realip_remote_addr` (the
original pre-rewrite source IP, preserved by the realip module
specifically for this case). The site config then enforces with
`if ($is_cloudflare = 0) { return 403; }`. Order of operations:

1. nginx accepts TCP connection from Cloudflare edge IP X.
2. realip module reads `CF-Connecting-IP: <visitor>` header and
   rewrites `$remote_addr = <visitor>`. Preserves
   `$realip_remote_addr = X`.
3. `geo $realip_remote_addr $is_cloudflare { ... }` evaluates
   against X → finds match → `$is_cloudflare = 1`.
4. Server block: `if ($is_cloudflare = 0)` → false → continue.
5. Request reaches Next.js / Laravel with `$remote_addr = <visitor>`
   so per-IP throttles work correctly.

## Deploy steps (run as root on the box)

1. **Copy files into place.**

   ```bash
   # From your laptop:
   scp infra/nginx/bin/refresh-cf-ips.sh         root@198.211.116.44:/usr/local/bin/
   scp infra/nginx/snippets/cloudflare-realip.conf root@198.211.116.44:/etc/nginx/snippets/
   scp infra/nginx/snippets/hsts.conf              root@198.211.116.44:/etc/nginx/snippets/
   scp infra/nginx/sites/app.bkrdy.me.conf        root@198.211.116.44:/etc/nginx/sites-available/app.bkrdy.me
   scp infra/nginx/sites/api.bkrdy.me.conf        root@198.211.116.44:/etc/nginx/sites-available/api.bkrdy.me
   scp infra/nginx/sites/tenant-wildcard.bkrdy.me.conf \
       root@198.211.116.44:/etc/nginx/sites-available/tenant-wildcard.bkrdy.me

   # On the server:
   chmod +x /usr/local/bin/refresh-cf-ips.sh
   mkdir -p /etc/nginx/snippets
   ```

2. **Edit `/etc/nginx/nginx.conf`** — inside the `http { ... }` block,
   add BOTH includes. `realip` first so it's defined before anything
   that references `$realip_remote_addr`:

   ```nginx
   include /etc/nginx/snippets/cloudflare-realip.conf;
   include /etc/nginx/snippets/cloudflare-geo.conf;
   ```

   Both lines are required. Missing the `realip` line is the trap we
   hit on 2026-06-09: the geo block still rejected direct-IP bypass
   attempts (so the surface looked locked), but `$remote_addr` was
   never rewritten, so Laravel's `Request::ip()` returned the
   Cloudflare edge IP instead of the visitor's real IP. Every per-IP
   throttle was keying on a tiny handful of CF edges — effectively
   lumping all real-world traffic together. Verify with the access-log
   probe in §Verification below.

3. **Run the refresh script once** to generate
   `/etc/nginx/snippets/cloudflare-geo.conf` (the file referenced by
   the second include you just added). The script also runs
   `nginx -t` and reloads, so it'll fail loudly if anything's off.

   ```bash
   /usr/local/bin/refresh-cf-ips.sh
   ```

4. **Confirm the site config TODOs.** Each `sites/*.conf` file has a
   `TODO when copying to the server:` block at the top — verify the
   PHP-FPM socket path, the wildcard cert path, and the Next.js port.

5. **Enable the sites.** (Disable any pre-existing default-server blocks
   first so the new ones aren't shadowed.)

   ```bash
   rm -f /etc/nginx/sites-enabled/default
   ln -sf /etc/nginx/sites-available/app.bkrdy.me            /etc/nginx/sites-enabled/
   ln -sf /etc/nginx/sites-available/api.bkrdy.me            /etc/nginx/sites-enabled/
   ln -sf /etc/nginx/sites-available/tenant-wildcard.bkrdy.me /etc/nginx/sites-enabled/
   nginx -t && systemctl reload nginx
   ```

6. **Install the cron entry** so the Cloudflare IP list stays current:

   ```bash
   cat > /etc/cron.d/refresh-cf-ips <<'EOF'
   0 4 * * 0 root /usr/local/bin/refresh-cf-ips.sh >> /var/log/cf-ip-refresh.log 2>&1
   EOF
   ```

   (Weekly is more than enough — Cloudflare adds new ranges maybe once
   a year.)

## Verification

After deploy, from your laptop:

```bash
# 1. Sanity — site still serves normally via Cloudflare.
curl -sI https://app.bkrdy.me/login | head -5
curl -sI https://api.bkrdy.me/api/v1/public/sites/lushstudio | head -5
# Both should be 200 / 404 respectively with Server: cloudflare + CF-RAY headers.

# 2. The fix — direct-IP bypass should now be blocked.
curl -sI -k -H "Host: app.bkrdy.me" \
  --resolve app.bkrdy.me:443:198.211.116.44 \
  https://app.bkrdy.me/login
# Should be: HTTP/1.1 403 Forbidden (or connection closed by nginx).
# Before this config: HTTP/1.1 200 OK with the full editor page.

curl -sI -k -H "Host: api.bkrdy.me" \
  --resolve api.bkrdy.me:443:198.211.116.44 \
  https://api.bkrdy.me/api/v1/public/sites/lushstudio
# Same — 403, not 404 from Laravel.

# 3. Real IP propagation — make a request through Cloudflare, then
#    check the laravel.log to confirm Request::ip() is your IP, not
#    a Cloudflare edge IP.
curl -s https://api.bkrdy.me/api/v1/auth/login -X POST \
  -H 'Content-Type: application/json' \
  -d '{"email":"bogus@example.com","password":"x"}'
ssh root@198.211.116.44 'tail -1 /var/www/bookready-api/api/storage/logs/laravel.log'
# Look for your laptop's public IP, not a 172.67.x or 104.21.x.
```

## Authenticated Origin Pulls (optional, recommended)

Belt-and-suspenders: even if the Cloudflare IP list ever falls out of
sync, AOP makes the origin reject any client that can't present
Cloudflare's edge certificate.

1. Cloudflare dashboard → SSL/TLS → Origin Server → Authenticated Origin
   Pulls → toggle on.
2. Download Cloudflare's Origin Pull CA cert and put it at
   `/etc/nginx/cloudflare-origin-pull-ca.pem`. The current download URL
   is documented at <https://developers.cloudflare.com/ssl/origin-configuration/authenticated-origin-pull/>.
3. In each `sites/*.conf` server block, uncomment the three
   `ssl_client_certificate` / `ssl_verify_client` / `ssl_verify_depth` lines.
4. `nginx -t && systemctl reload nginx`.
5. Re-run the verification curls above. The bypass attempts will now
   fail at the TLS handshake instead of at the access-control layer
   (good — earlier failure = smaller attack surface).

## After all this is up

1. **Rotate the origin IP** at some quiet moment. The current IP is
   permanently in DNS history and CT logs. DigitalOcean lets you swap
   to a floating IP — deploy to the new IP, repoint Cloudflare, retire
   the old one. The bypass attack only works if the attacker knows the
   IP, so rotating is a forcing function to invalidate the leaked one.
   Not urgent given the whitelist + AOP, but worth doing at some point.
2. **Check censys.io / shodan.io for `198.211.116.44`** — they index
   the origin's certificate Subject Alternative Names, which is how
   attackers usually find an origin even when it's behind a CDN. If
   your origin cert SANs say `app.bkrdy.me`, Shodan will surface it.
   Cloudflare's Origin Server certs (separate from Let's Encrypt) don't
   leak the customer-facing hostnames; consider switching to one of
   those after AOP is enabled.
